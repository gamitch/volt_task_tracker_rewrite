VOLT Team Portal — Product Backlog & Grooming Guide

Written 2026-07-19 by the orchestrator session that built the app (swarm branch claude/swarm-plan-zl575z, PR #1). Audience: a future model session doing backlog grooming and task assignment, and George (product owner). This document is deliberately information-dense so a smaller model can act on it without re-deriving context.

0. How to use this document (instructions to the grooming model)

  1. Read the ground-truth sections (1–3) before touching any backlog item. They encode org facts and codebase facts that are NOT discoverable from the code alone.
  2. Follow the existing swarm process for any implementation work. Read docs/swarm/constitution.md, docs/swarm/task-ledger.md, and the packet format in docs/swarm/active/EXAMPLE-worker-packet.md. Every task gets a scoped worker packet (Allowed Files / Forbidden Files / Traps / Acceptance Criteria) and an independent checker; no worker self-certifies. Continue task IDs from where the ledger leaves off.
  3. Constitution rules that most often bite new work (all BLOCKER-class): database migrations are additive-only (never edit a shipped migration file); metric formulas live in SQL views only, never re-derived in TypeScript; no secrets or service-role keys anywhere in the frontend or committed files; no PII in URLs/logs/fixtures (fabricated names only in tests); UI is Astryx design system only (verify props against docs/swarm/astryx-api.md or the installed source under node_modules/@astryxdesign/core/src/ — never guess); motivation mechanics limited to honest progress signals — no streaks, FOMO, countdowns, guilt copy, or re-engagement hooks; users are minors.
  4. Ask George before acting on anything marked "Open product questions." Do not resolve product ambiguity unilaterally.
  5. Sizing legend: S = one small packet; M = 2–5 packets; L = an epic (5–15 packets); XL = multi-epic, needs boss-architect design pass first.

2. Codebase ground truth (verified in-session, 2026-07-19)

  • Stack: Vite + React 18 + TypeScript strict; Astryx design system (no Tailwind/shadcn, per PRD D3); Supabase (Postgres + RLS + Edge Functions: checkin, ics, send-invite, send-reminders); Resend for email (sender notifications@mail.voltfrc.org, domain verified); Vitest + Playwright.
  • Roles: Postgres enum role_enum = 'admin' | 'coach' | 'student' | 'parent'. No mentor, no student-coach. (Postgres ALTER TYPE ... ADD VALUE is additive and allowed.)
  • Key schema facts (from supabase/migrations/, 7 files, all applied to the production project):
    • students.team_id is a single required FK — one team per student. Blocker for dual membership (item B2).
    • seasons has a partial unique index allowing exactly one active season globally. Blocker for overlapping FTC/FRC/FLL seasons (item B3).
    • teams.program is a check (program in ('FRC','FTC','Other')) constraint — no 'FLL' value. Check constraints can't be "extended" additively; an additive workaround (new constraint replacing old via add-new-drop-old in one migration, or classifying FLL as 'Other') needs an explicit decision. Affects item B4.
    • ROS-08's leaderboard-privacy toggle has no persistence column (UI works against an injectable seam; T028's checker logged the follow-up: likely seasons.leaderboard_show_full_name boolean not null default true).
    • Metric views v_student_participation / v_student_hours are the only allowed source of metric math (constitution item 3).
  • Deployment state: production Supabase project exists; migrations applied; all four Edge Functions deployed; Resend domain verified; RESEND_API_KEY + cron Vault secrets set. As of this writing RESEND_SEND_MODE is not yet flipped to production (T052 human gate, in progress). No Vercel deployment yet (T070 blocked). First admin account created manually (auth user + hand-inserted profiles row).
  • Ledger state: 78 tasks Passed; human gates T052 (email), T063–T065 (data migration chain), T070 (domain go-live) remain. T085 (shell wiring, see ED-0) in flight at time of writing.
⚠ THE SINGLE MOST IMPORTANT ENGINEERING FACT (ED-1, P0): nearly every page still uses its fixture-backed default data loader. The pages were built with injectable loadData seams (correct architecture), but the follow-up tasks to wire those seams to the real Supabase data layer (src/lib/supabase/, T020) were never scoped or dispatched — T020's own packet explicitly declared them out of scope ("one small follow-up per page or per epic") and no such tasks were ever created. Consequence: the production app renders fabricated fixture data (e.g. the dashboard's "82.4% participation", "Priya Patel" signups) even when logged in against the real, empty production database. Mutations are affected too: e.g. InvitesTab's resend/revoke default to fixture-backed fakes; sending a real invite through the UI will not hit the real send-invite Edge Function until wired. Every feature below is cosmetic until ED-1 is done. It also blocks completing T052's email smoke test through the UI.

4. Product backlog (priority order)

B1 — Wire the real data layer (= ED-1)P0L

Listed here as well as in §3 because it is the true top of the product backlog: nothing below it delivers user value until the app shows real data. See the §2 callout for full detail. Groom this first.

B2 — Multi-team student membershipP0XL

Problem
P3/Gear Girls students can also be VOLT members; students.team_id (single FK) cannot express it. Workaround today would be duplicate student rows — which would double-count hours and fracture attendance history.
Direction
Additive junction table (e.g. student_teams(student_id, team_id, joined_on, left_on)), backfilled from team_id; keep team_id as legacy/primary-team until all readers migrate. Touches: attendance expectations (which team's meeting counts for whom), per-team vs. combined hours credit, leaderboard scoping, reports filters, roster UI (team chips instead of single team column), invite flow.
Open product questions
Do hours earned at a P3 meeting count toward that student's VOLT totals, or are hours strictly per-team? Is there a "primary" team for display purposes? Can parents of a dual-membership student see both teams' calendars (presumably yes)?
Sequencing
Needs boss-architect design pass. Do BEFORE significant real data accumulates (migration cost grows with data). Depends on B1 (pointless while pages show fixtures).
Acceptance seed
A student can belong to ≥2 teams; appears on both rosters; attendance at either team's session credits correctly per the answer to the hours question; leaderboard/report totals do not double-count.

B3 — Per-program (or per-team) seasonsP0L

Problem
DB enforces exactly one active season globally; FTC, FRC, and FLL calendars overlap. Today, activating the FRC season would deactivate the FTC season mid-competition.
Direction
Additive: add a scope column to seasons (e.g. program or team_id), replace the single partial unique index with a per-scope one (new index + drop old in one additive migration — index changes are not "editing a shipped migration"). Season-picker UI follows; note every report/tab currently receives one seasonId prop — the threading exists, only the selection source changes.
Open product questions
Scope seasons per program (one FTC season shared by P3 + Gear Girls) or per team? Per-program is simpler and matches FIRST's structure; per-team allows different goal-hours per team — but students.goal_hours_override already handles individual variance.
Sequencing
Decide alongside B2 (both are "before real data accumulates" schema decisions). Same boss-architect pass can cover both.

B4 — Student-coach role + FLL team supportP1XL

Problem
Five FLL teams are coached by P3/Gear Girls students at external venues on their own schedules. None of this is in the app: FLL isn't a valid teams.program value, students can't run check-in or manage a team, and the coaching hours (structured outreach, Impact Award gold) go untracked.
Direction
(a) Add FLL as a program value (see §2 for the check-constraint trap). (b) New coaching_assignments table (student ↔ team they coach). (c) A capability model: a student-coach gets coach-like powers scoped to their assigned FLL team only (run check-in/kiosk, mark sessions complete) without global coach powers — this is a real authorization design decision (RLS policies must enforce the scoping, not just UI). (d) Auto-credit: an FLL session run by a student-coach credits that student outreach/volunteer hours on their own team's ledger.
Open product questions
Do FLL team members themselves get accounts (they're likely too young — COPPA/consent concern; probably the FLL teams exist as teams with sessions but WITHOUT student-member accounts, and only attendance counts are recorded, no rosters)? Who confirms a student-coach's hours — an adult coach approval step?
Traps
Constitution: users are minors, no PII — FLL kids' names should probably never enter the system at all. RLS changes are BLOCKER-class review territory; checker must verify policies with a real scratch-Postgres harness like T012/T014 did (supabase/tests/ has the precedent).
Sequencing
After B2/B3 (it builds on team-scoping and per-program seasons). Boss-architect pass required for the authorization model.

B5 — Mentor role + central mentor hoursP1L

Problem
Non-coach adult mentors track hours in a separate system; George wants it centralized. Grant applications and Impact Award submissions ask for mentor-hour numbers.
Direction
ALTER TYPE role_enum ADD VALUE 'mentor' (additive-safe). Mentors check in via the existing kiosk flow (machinery exists); appear in a mentors section of roster; hours roll up per team/season in reports. Decide what mentors can SEE (probably: their own hours + team calendar; NOT student PII beyond first names — SEC-02 posture). RLS policies needed for the new role everywhere (audit every existing policy that enumerates roles — is_staff() etc. — for whether mentor belongs).
Open product questions
Do mentors self-register or get invited like parents? Do mentor hours need coach approval, or is kiosk check-in trusted?
Acceptance seed
A mentor can be invited, check in at a meeting, and a coach can export "mentor hours by team for season X".

B6 — Venues on meetings/eventsP1S

Problem
Five regular venues (garage, basement, O'Fallon Public Library, United Methodist Church, competition sites); events have no location field; parents can't see where to go from calendar feeds or reminder emails.
Direction
Additive events.location (or a small venues table if reuse is wanted); surface in event dialogs, detail pages, the ICS feed (LOCATION property — supabase/functions/ics/location.ts already exists as the seam), and reminder-email templates. Cheap, immediately useful, no schema controversy.
Sequencing
Can go anytime; a good first packet for a new grooming session to calibrate on. Best right after B1 so it's visible with real data.

B7 — Parent volunteer signupsP2M

Problem
Parents can only view + RSVP for their student. Build nights need food, competitions need transport/chaperones — coordinated off-app today.
Direction
Volunteer-slot concept on events (slot name, count needed, sign-up), reusing the outreach signup mechanics/UI patterns. Parent home page gets a "help needed" section. Reminder emails include committed slots.
Open product questions
Slots freeform per event, or a reusable catalog (food/transport/chaperone)? Can students sign up for volunteer slots too?

B8 — Impact Award / grant reporting exportP2M

Problem
Outreach data (events, hours, signups — plus B4's student-coaching and B5's mentor hours once they exist) is exactly what FIRST Impact Award essays and grant applications need, but there's no aggregated export.
Direction
A report tab or export: season-scoped totals (outreach events run, unique students participating, student outreach hours, people-reached if captured, mentor hours, FLL-coaching sessions). CSV export precedent exists (src/pages/reports/csvExport.ts). Metric math in SQL views only (constitution item 3) — new aggregates mean new additive views.
Sequencing
Value multiplies after B4/B5 land; a minimal version (existing outreach data only) can ship anytime after B1.

B9 — Program-level cross-team dashboardP2M

Problem
George (admin over the whole program) sees one team at a time; no rollup of participation/hours/outreach across VOLT + P3 + Gear Girls (+ FLL later).
Direction
An admin-only view aggregating the existing per-team metrics across teams, season-scoped. Depends on B3's season model to be meaningful across programs. Read-only; low schema impact (new views only).

B10 — Build-season task boardP2XL

Problem
The app tracks who showed up but not what got done. Subsystem/task tracking (owners, due dates, status) would connect attendance to output. Ironically the repo is named "task tracker" and this is the one thing it doesn't do.
Direction
Genuinely new domain (tasks, assignments, statuses) — needs its own boss-architect design pass and PRD-style spec from George before any grooming. Resist scope creep toward a full PM tool; the value is "what is blocking the robot this week," not Jira.
Constitution note
Task-assignment UI must stay within motivation-ethics rules: no deadline-pressure mechanics aimed at students, no public shame boards. Progress signals must stay honest and neutral.

B11 — Tool/safety certification trackingP3M

Direction
Per-student certification records (tool, certified-on, certified-by), coach-managed, surfaced on the roster student detail. Simple additive table + roster UI. Useful for shop safety audits.

B12 — Announcements feedP3M

Direction
Coach-posted announcements, role/team-targeted, shown on home pages, optionally included in the existing weekly digest email rather than a new send (reuses T051's infrastructure; keeps email volume down). Read-receipts only if genuinely needed — that's surveillance-adjacent for minors; default to not tracking reads.

B13 — Inventory / purchase requests / budgetP3XL

Direction
Parts inventory, purchase-request workflow, per-season budget tracking. Big, separate domain; validate demand before designing (does George actually want this in-app, or is a spreadsheet fine?). Do not groom without an explicit go-ahead.

B14 — Season-over-season analytics + alumni trackingP3L

Direction
Retention rate, participation by grad-year cohort, alumni list (for mentor pipeline — feeds B5). Coach/admin-facing only. Early-disengagement signals go to coaches, never to students (motivation-ethics rule). Needs ≥2 seasons of real data to be worth building — schedule accordingly.

B15 — Sponsor managementP3M

Direction
Sponsor list, contact log, deliverables owed (logo placements, demo visits, thank-yous), per-season. Adult-facing only; zero student involvement, so low constitution risk. Validate demand first.

5. Cross-cutting grooming notes

  • Recommended sequencing spine: B1 (real data) → B2+B3 together (schema decisions, one boss-architect pass) → B6 (quick win) → B5 → B4 → B8 → the rest by demand. Human gates T052/T063–T065/T070 proceed in parallel with George.
  • Anything touching RLS or roles must be checker-verified against a real scratch Postgres (precedent: supabase/tests/run.sh, T012/T014 methodology in the verification log), not just unit tests.
  • Email changes stay behind the fail-closed RESEND_SEND_MODE mechanism; test mode is the default everywhere; production sending is a George-only decision (constitution item 16).
  • When in doubt about a product question, stop and ask George. The Open-product-questions lists above are genuinely open — none have been answered yet unless a newer document says otherwise.
